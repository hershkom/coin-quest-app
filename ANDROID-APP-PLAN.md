**עדכון 2026-07-19 — בוצעו S1, S2, S3 (חסמי הפרסום הראשונים):**
- **S1 (כללי אבטחה בגיט):** נוצר `database.rules.json` (auth חובה + התאמת `users/{uid}.familyId` ל-`families/$familyId`, `inviteCodes` קריא לכל מחובר אך נכתב רק להצביע למשפחה של הכותב) וחובר ל-`firebase.json` (`"database":{"rules":"database.rules.json"}`) כדי שיהיה מנוהל בגיט ונפרס עם `firebase deploy --only database` — לפני כן הכללים היו רק בקונסולה, לא בריפו. **פעולה נדרשת מהמשתמש:** להריץ בפועל `firebase deploy --only database` כדי לפרסם את הקובץ הזה (לא הורצה אוטומטית בסשן זה). גם עודכנה הערה מיושנת ב-`app.js` שאמרה "open test-mode".
- **S2 (חיזוק PIN):** נוספה `isWeakPin()` שחוסמת קודי הורים טריוויאליים (ספרה חוזרת כמו 0000/1111, רצף כמו 1234/4321) בשני מקומות: `savePin()` (הגדרות הורה) ו-`finishWizard()` (הקמת משפחה חדשה). גם שונה placeholder השדה באשף מ-"לדוגמה: 1234" ל"קוד לא מובן מאליו".

## חלק א' — שלבי פיתוח האפליקציה לאנדרואיד (AN1–AN10)

### AN1. הקראה קולית (TTS) אמינה בתוך ה-WebView — קול עברית נייטיבי
**בעיה:** ה-Web Speech API בתוך WebView של אנדרואיד לא תמיד מספק קול עברית (תלוי בחבילת
קולות מותקנת), וההקראה החדשה (`speakWithHighlight` ב-`app.js`) עלולה להיות אילמת במכשיר.
**מה לעשות:**
1. להוסיף ל-`NativeGameBridge.kt` (או קובץ חדש `NativeTtsBridge.kt`) מנוע TTS נייטיבי:
   `android.speech.tts.TextToSpeech` עם `Locale("he","IL")`, ולחשוף לג'אווהסקריפט שתי
   פונקציות: `@JavascriptInterface fun speak(text: String, lang: String, utteranceId: String)`
   ו-`@JavascriptInterface fun stopSpeak()`.
2. להאזין ל-`UtteranceProgressListener` — ב-`onRangeStart(utteranceId, start, end, frame)`
   (זה המקביל הנייטיבי ל-boundary event, נותן אינדקסים של המילה הנוכחית!) לקרוא חזרה
   ל-JS: `webView.evaluateJavascript("window._nativeTtsBoundary&&window._nativeTtsBoundary('${utteranceId}',${start},${end})", null)`
   (על ה-UI thread — לעטוף ב-`webView.post { ... }`). ב-`onDone`/`onError` לקרוא
   `window._nativeTtsEnd('${utteranceId}')`.
3. ב-`app.js`, בתוך `speakWithHighlight()`: אם `window.CoinQuestNative?.speak` קיים —
   להשתמש בו במקום `speechSynthesis` (אותה לוגיקת הדגשת מילים, רק שמקור האירועים הוא
   `_nativeTtsBoundary`/`_nativeTtsEnd` במקום `onboundary`/`onend`). לשמור על מונה
   `_ttsGen` בדיוק כמו היום. `stopSpeaking()` יקרא גם ל-`CoinQuestNative.stopSpeak()`.
4. אם אין קול עברית מותקן (`tts.isLanguageAvailable(...) < 0`) — להחזיר מ-`speak` ערך
   שגיאה, וב-JS ליפול חזרה ל-`speechSynthesis` ואז ל-safety-net timeout הקיים.
**בוצע כאשר:** במכשיר אמיתי שאלה בלמידה מוקראת בעברית עם הדגשת מילים חיה, ותשובה באנגלית
מוקראת בקול אנגלי.

### AN2. אייקון ומסך פתיחה (Splash) מותאמים
**מה לעשות:**
1. להחליף את `ic_launcher_foreground.xml` באייקון של כספת+מטבע בסגנון האפליקציה (אפשר
   וקטור פשוט: כספת סגולה `#7C4DFF` עם מטבע זהב `#FFC107`). לשמור adaptive icon.
2. להוסיף Splash API של אנדרואיד 12+: תלות `androidx.core:core-splashscreen:1.0.1`,
   theme `Theme.SplashScreen` עם `windowSplashScreenBackground` בצבע הרקע של האפליקציה
   ו-`windowSplashScreenAnimatedIcon` = האייקון. ב-`MainActivity.onCreate` לקרוא
   `installSplashScreen()` לפני `setContentView`. זה מסתיר את שנייה-שתיים של טעינת ה-WebView.
**בוצע כאשר:** פתיחת האפליקציה מציגה splash נייטיבי חלק שמתמזג עם מסך הפתיחה הקיים של הווב.

### AN3. מצב אופליין ידידותי לילד
**בעיה:** בלי רשת ה-WebView מציג דף שגיאה טכני באנגלית — מלחיץ לילד אוטיסט.
**מה לעשות:**
1. ב-`MainActivity`, ב-`WebViewClient.onReceivedError` (רק כש-`request.isForMainFrame`):
   לטעון קובץ מקומי `file:///android_asset/offline.html`.
2. ליצור `android-app/app/src/main/assets/offline.html`: עמוד RTL בעברית פשוטה עם אימוג'י
   גדול (📡), טקסט "אין אינטרנט כרגע — נסו שוב עוד רגע", וכפתור ענק "נסה שוב" שקורא
   `location.href='https://coin-quest-app.firebaseapp.com/'`. צבעים ופונט כמו האפליקציה.
3. בנוסף, להרחיב את `sw.js` הקיים כך שיעשה cache ל-`index.html`, `app.js`, `styles.css`,
   `questions.js` (אסטרטגיית stale-while-revalidate) — כך האפליקציה נפתחת גם בלי רשת,
   והסנכרון מתעדכן כשחוזרת.
**בוצע כאשר:** כיבוי WiFi + פתיחת האפליקציה מציג עמוד עברי ידידותי או את האפליקציה מה-cache.

### AN4. נעילת קיוסק רכה (הילד לא יוצא בטעות)
**מה לעשות:**
1. ב-`onBackPressed`: כשאין היסטוריית WebView, במקום לצאת מיד — להציג דיאלוג פשוט בעברית
   "לצאת מהאפליקציה?" עם כפתורים גדולים ✔/✖ (או: לחיצה כפולה על חזרה תוך 2 שניות יוצאת,
   לחיצה בודדת מציגה toast "לחצו שוב כדי לצאת").
2. להוסיף טיפול ב-swipe-refresh: היום כל גרירה למטה מרעננת את העמוד ומאבדת מצב משחק/שאלה.
   לכבות את `swipeRefresh` כברירת מחדל ולהפעיל רענון רק דרך כפתור בהגדרות ההורה
   (`webView.evaluateJavascript` יכול לחשוף `CoinQuestNative.setPullToRefresh(enabled)` —
   פשוט יותר: `swipeRefresh.isEnabled = false` ולהוסיף לגשר פונקציה `reload()`).
**בוצע כאשר:** גרירה למטה באמצע שאלה לא מאפסת אותה, ויציאה מהאפליקציה דורשת פעולה מכוונת.

### AN5. התראות מקומיות: תזכורת מטלות ו"נגמר הזמן בקרוב"
**מה לעשות:**
1. בגשר הנייטיבי להוסיף `@JavascriptInterface fun scheduleReminder(hour: Int, minute: Int, text: String)`
   באמצעות `AlarmManager.setInexactRepeating` + `BroadcastReceiver` שמציג נוטיפיקציה
   (ערוץ נוטיפיקציות "תזכורות", חשיבות DEFAULT — בלי צליל מבהיל, ראו חלק ג').
2. באפליקציית הווב, בהגדרות ההורה: שדה שעה "תזכורת מטלות יומית" שקורא לפונקציה הזו.
3. באכיפת זמן המשחק הקיימת (`GameTimeOverlayService`): 2 דקות לפני סוף הזמן להציג
   נוטיפיקציה + טקסט על ה-overlay "עוד 2 דקות ⏳" — התרעה מוקדמת חשובה במיוחד לילדים
   אוטיסטים (מעבר פתאומי = התמוטטות; ראו A2 בחלק ג').
**בוצע כאשר:** נוטיפיקציית תזכורת מופיעה בשעה שנקבעה, והתרעת "2 דקות" מופיעה לפני סוף משחק.

### AN6. שמירת מסך דולק בזמן שאלה/משחק
**מה לעשות:** בגשר: `keepScreenOn(on: Boolean)` שמפעיל/מכבה
`window.addFlags(WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON)` (דרך `runOnUiThread`).
ב-`app.js`: להדליק בכניסה ל-view של למידה/משחק (`go('learn')`), לכבות ביציאה.
**בוצע כאשר:** המסך לא נכבה באמצע שאלה גם אחרי timeout של המערכת.

### AN7. גרסת release חתומה + הקשחה
**מה לעשות:**
1. ליצור keystore (`keytool -genkeypair ... -validity 10000`), לשמור **מחוץ לריפו**,
   ולהוסיף `signingConfigs.release` ב-`app/build.gradle` שקורא סיסמאות מ-`local.properties`
   או ממשתני סביבה (ב-CI: GitHub Secrets + שלב `assembleRelease`).
2. להשאיר `minifyEnabled false` בשלב ראשון (לקח מפרויקט tap_runner: R8 שבר build שם;
   אם מפעילים — חובה בדיקת sideload לפני העלאה).
3. `android:allowBackup="false"` במניפסט (שלא ישחזרו נתוני PIN ממכשיר אחר),
   `versionCode`/`versionName` מנוהלים ידנית בכל שחרור.
**בוצע כאשר:** `./gradlew assembleRelease` מפיק APK חתום שמותקן ורץ במכשיר אמיתי.

### AN8. עדכון-עצמי פשוט (בלי חנות)
כי ההפצה היא sideload משפחתי, לא Google Play:
1. בכל build של CI לכתוב קובץ `version.json` (`{"versionCode": N, "apkUrl": "..."}`)
   ל-GitHub Releases (או לתיקיית ההפצה של Firebase Hosting).
2. בגשר: `getVersionCode()`. ב-`app.js`, פעם ביום: fetch ל-`version.json`, ואם גרסת
   הענן גדולה — להציג להורה (רק במסך ההורה!) באנר "יש עדכון לאפליקציה" עם קישור להורדה.
**בוצע כאשר:** העלאת גרסה חדשה מציגה באנר עדכון במכשיר תוך יום.

### AN9. בדיקות במכשיר אמיתי — צ'קליסט חובה לכל שחרור
לרוץ ידנית לפני כל APK שנמסר לילדים:
- [ ] כניסת PIN הורה עובדת; כניסת ילד עם לחיצה אחת.
- [ ] סימון מטלה → אנימציית מטבע → יתרה עולה; סימון שני תוך דקה נחסם (cooldown).
- [ ] קניית זמן משחק → הזמן מופיע בבאנר → נכנס למשחק → הטיימר יורד → הזמן נאכף (overlay).
- [ ] שאלה בלמידה מוקראת בקול עם הדגשת מילים; כפתור 🔊 חוזר על ההקראה; תשובה עוצרת הקראה.
- [ ] סריקת QR במצלמה עובדת (הרשאה + פוקוס).
- [ ] מכשיר שני: שינוי בהגדרות הורה מסתנכרן תוך שניות.
- [ ] מצב טיסה: עמוד אופליין ידידותי; חזרת רשת: סנכרון מתאושש.
- [ ] סיבוב מסך/מעבר לאפליקציה אחרת וחזרה: המצב לא אבד.

## חלק ב' — שיפורי נראות: גרפיקה אטרקטיבית לילדים (G1–G8)

> ממשיכים את V1–V10 מ-DESIGN-IMPROVEMENTS.md (בוצעו). כל השיפורים חייבים לכבד את
> `prefers-reduced-motion` ואת דגלי ההרגעה של חלק ג'.

### G1. דמות מלווה (מסקוט) לכל ילד
דמות SVG קטנה וקבועה שמופיעה בפינת המסך: לאריאל — קריפר/גולם ידידותי בסגנון בלוקים
(בלי קווים חדים/מפחידים); לנועה — דמות מהעולם שלה. הדמות מגיבה באירועים: תשובה נכונה →
קופצת (CSS `transform: translateY` keyframes, 400ms); סימון מטלה → מוחאת כפיים (החלפת
frame); חוסר פעילות 30 שניות במסך למידה → מנפנפת. מימוש: `<div id="mascot">` קבוע
ב-`index.html`, פונקציה `mascotReact(kind)` ב-`app.js`, שני קבצי SVG inline לפי theme.
**חשוב:** הבעות פנים פשוטות וחד-משמעיות בלבד (שמח/רגוע) — לא הבעות מעורפלות.

### G2. פס התקדמות יומי חזותי ("מפת מסע")
במקום מספרים בלבד: שביל אופקי עם 5–8 תחנות (עיגולים), כל מטלה שהושלמה צובעת תחנה ומזיזה
את דמות הילד תחנה קדימה. בסוף השביל — תיבת אוצר שנפתחת (אנימציית CSS) כשכל המטלות הושלמו.
מימוש בתוך `renderDayStrip()` הקיים; תחנות = `flex` של spans, הדמות = אימוג'י/SVG קטן
עם `transition: right .6s ease`.

### G3. חנות פרסים ויזואלית עם "מד חיסכון"
לכל פרס בחנות: עיגול התקדמות (SVG `stroke-dasharray`) שמראה כמה מהמחיר כבר יש לילד —
"יש לך 70 מתוך 100 🪙". פרס בר-השגה מקבל זוהר זהוב עדין (`box-shadow` פועם — סטטי אם
reduced-motion). זה הופך חיסכון ממושג מופשט לתמונה — קריטי לגיל 7 ולחשיבה קונקרטית.

### G4. אווטאר מטבע מתפתח
היתרה מוצגת עם ערימת מטבעות מצוירת שגדלה במדרגות (1–3 מטבעות / שקית / תיבה / כספת
זוהרת לפי גובה היתרה). מימוש: פונקציה `coinPileFor(balance)` שמחזירה אחד מ-4 SVG,
מוחלפת ב-`renderBalance()` עם `crossfade` קצר.

### G5. חגיגות מדורגות (לא מבהילות)
שלוש רמות חגיגה: תשובה נכונה = ✔ ירוק + צליל רך קצר; השלמת סשן למידה = גשם קונפטי עדין
2 שניות; שיא אישי חדש = מסך "כל הכבוד!" עם הדמות המלווה. **אף פעם לא:** הבזקים,
רעידות מסך, צלילים חזקים פתאומיים. עוצמת החגיגה נשלטת בדגל `state.settings.celebrationLevel`
(ראו A6).

### G6. ערכת אייקונים אחידה למטלות
לכל מטלה אימוג'י/אייקון גדול (48px+) שההורה בוחר מרשימה סגורה בעת יצירת המטלה
(🦷🛏️🎒📚🍽️🧸🚿...). ילד שלא קורא מזהה מטלה לפי תמונה. שדה `icon` באובייקט המטלה,
בורר אייקונים בטופס ההורה, תצוגה ב-`renderChores()`.

### G7. רקעים חיים לפי שעת יום
הרקע של מסך הבית משתנה עדין לפי שעה: בוקר (שמיים בהירים), צהריים, ערב (שקיעה), לילה
(כוכבים) — עוזר גם לעיגון בזמן (ילדים אוטיסטים נשענים על עוגני שגרה). מימוש: 4 מחלקות
CSS gradient על `body`, נבחרות ב-`applyTheme()` לפי `new Date().getHours()`.

### G8. משוב מגע (haptic) עדין
`navigator.vibrate(20)` (או דרך הגשר הנייטיבי) בסימון מטלה ובתשובה נכונה — רטט קצרצר
אחד, לא תבניות. כבוי כברירת מחדל אם `state.settings.calmMode` פעיל, וניתן לכיבוי בהגדרות.

## חלק ג' — שיפורי לוגיקה מותאמים לילדים אוטיסטים (A1–A8)

> עקרונות מנחים: צפוּיוּת (predictability), מעברים מוכנים מראש, עומס חושי מינימלי,
> אפס ענישה, שפה קונקרטית וחיובית.

### A1. "עכשיו → אחר כך" בכל מסך
בכל רגע הילד רואה בדיוק שני דברים: מה עושים עכשיו ומה יקרה אחר כך ("עכשיו: 3 שאלות
חשבון → אחר כך: 10 דקות משחק"). להרחיב את `renderFirstThen()` הקיים כך שיופיע כרצועה
קבועה בראש **כל** מסך ילד, לא רק במסך הבית, עם שני כרטיסים מחוברים בחץ.

### A2. טיימרים חזותיים + התרעות מעבר מדורגות
כל ספירה לאחור (זמן משחק, זמן לשאלה) מוצגת כעיגול מתרוקן ויזואלי, ולפני כל מעבר יש
שלוש התרעות עדינות: 5 דקות ("עוד מעט מסיימים"), 2 דקות, 30 שניות — טקסט + הדמות המלווה
מצביעה על השעון. אף פעם לא ניתוק פתאומי: בסוף הזמן מסך "הזמן נגמר, היה כיף! 👋" ל-10
שניות לפני החזרה למסך הבית (ה-overlay הנייטיבי הקיים יקבל את אותה התנהגות).

### A3. שגרה קבועה ולוח יומי צפוי
מסך "היום שלי": רשימת המטלות תמיד באותו הסדר (ההורה קובע סדר, לא מיון אוטומטי!),
מטלה שהושלמה לא נעלמת אלא מקבלת ✔ גדול ונשארת במקומה — היעלמות פריטים שוברת צפיות.
שדה `order` למטלות + drag-to-reorder בטופס ההורה בלבד.

### A4. אפס ענישה, רק חיזוק
לוודא שאין בשום מקום ירידת מטבעות/נקודות אוטומטית: תשובה שגויה בלמידה = "בוא ננסה
שוב 💪" + הצגת התשובה הנכונה עם הקראה, בלי X אדום גדול ובלי צליל שלילי. אחרי 2 טעויות
באותה שאלה — השאלה "מתחלפת" (לא נכשלת) ומופיעה שוב מאוחר יותר קלה יותר (ראו A5).

### A5. הסתגלות קושי הדרגתית ושקטה
מנגנון פשוט ב-`answerLearningQuestion`: רצף 3 נכונות → עולים תת-רמה; 2 שגיאות → יורדים
תת-רמה, **בלי להודיע לילד** (שינוי רמה מוצהר = תסכול). לשמור `state.learning.adaptiveLevel`
פר-נושא פר-ילד. ההורה רואה את הרמה בפועל במסך ההורה בלבד.

### A6. "מצב רגוע" (Calm Mode) — מתג הורה אחד
דגל `state.settings.calmMode` שמכבה בבת אחת: קונפטי, רטט, צלילי חגיגה, אנימציות רקע
חיות, ומאט את קצב ההקראה (`utter.rate=0.8`). שקול ל-`prefers-reduced-motion` אבל ברמת
האפליקציה ונשלט על ידי ההורה — יש ימים רגישים. כפתור מהיר במסך ההורה + סנכרון בין מכשירים.

### A7. הכנה למעברים בין מסכים
מעבר בין views לא קורה מיידית בלחיצה על משהו שמסיים פעילות: קודם מסך ביניים קצר (1.5
שניות) "מסיימים... עוברים ל..." עם אייקון היעד. מימוש: wrapper ל-`go(v)` —
`goWithTransition(v, label)` שמציג overlay קצר. חל רק על מעברים "גדולים" (סיום למידה,
סיום משחק), לא על ניווט רגיל.

### A8. תקשורת חזותית עקבית — מילון סמלים אחיד
לקבע: 🪙 תמיד מטבעות, ⏳ תמיד זמן, ✔ ירוק תמיד הצלחה, 🎁 תמיד פרס — ולעבור על כל
המסכים לוודא שאין סמל שמשמש לשני דברים או שני סמלים לאותו דבר. בנוסף: כל כפתור פעולה
של ילד = אייקון + מילה אחת בלבד (לא משפטים), גודל מגע מינימלי 48px, מרווח בין כפתורים
שמונע לחיצות שגויות.

## חלק ד' — התאמות חובה לפרסום ב-Google Play (S1–S8)

> **עקרון העלויות:** אין צורך לשלם לאף גורם שלישי. חשבון Firebase נשאר בתוכנית **Spark
> (חינם)** — Auth, Realtime DB (עד 1GB / 100 חיבורים בו-זמנית) ו-Hosting כולם בחינם.
> הרשמת משתמשים כבר ממומשת בתוך האפליקציה (Google Sign-In + familyId + קודי הזמנה,
> `app.js` ~3583) — אין תלות בשירות רישום חיצוני. העלות היחידה: 25$ חד-פעמי לחשבון
> מפתח ב-Play. חלופה של "הכל מקומי במכשיר בלבד" הייתה מוותרת על סנכרון בין מכשירי
> ההורים והילד — לכן נשארים עם Firebase החינמי.
>
> **חסם קנה-מידה לידיעה בלבד:** מגבלת 100 חיבורים בו-זמנית ב-Spark מספיקה לעשרות-מאות
> משפחות פעילות. אם האפליקציה תצליח מעבר לזה — שדרוג ל-Blaze (תשלום לפי שימוש) הוא
> החלטה עתידית, לא שינוי ארכיטקטורה.

### S1. ביקורת Security Rules — לוודא ולתעד, לפני כל שיווק
כללים סגורים כבר נפרסו ב-2026-07-01 (auth חובה + התאמת `users/{uid}.familyId`), אבל:
(א) הם קיימים רק בקונסולת Firebase, לא בריפו — להוריד אותם לקובץ `database.rules.json`
ולחבר ל-`firebase.json` כדי שיהיו מנוהלים בגיט ונפרסים עם `firebase deploy`;
(ב) יש הערה מיושנת ב-`app.js` ~3447 שעדיין אומרת "open test-mode" — לעדכן;
(ג) לבצע ביקורת אימות מחדש לפני שיווק. הכללים הנדרשים:
- `families/$familyId/**`: קריאה/כתיבה רק אם `root.child('users/'+auth.uid+'/familyId').val() === $familyId`.
- `users/$uid`: קריאה/כתיבה רק אם `auth.uid === $uid` (יצירה ראשונית: כתיבה מותרת רק ל-uid עצמו).
- `inviteCodes/$code`: קריאה לכל משתמש מחובר (`auth != null`), כתיבה רק ע"י חבר במשפחה שהקוד מצביע עליה.
- ברירת מחדל: הכל חסום.
לפרוס דרך `firebase deploy --only database`, ולבדוק: מכשיר של משפחה א' לא מצליח לקרוא
נתוני משפחה ב' (בדיקה ידנית עם שני חשבונות Google).
**בוצע כאשר:** בקשת קריאה ל-familyId זר מחזירה permission_denied, וכל הבדיקות הקיימות עוברות.

### S2. חיזוק PIN הורים
משפחות חדשות כבר בוחרות PIN משלהן ב-onboarding (נוסף 2026-07-01) — הבעיה נותרה רק
בנתוני המשפחה המקורית. לגרסה ציבורית: (א) לוודא שה-PIN נשמר כ-hash (SHA-256 עם salt
אקראי פר-משפחה, `crypto.subtle.digest`) ולא בטקסט גלוי בסנכרון; (ב) לחסום PIN-ים
טריוויאליים ביצירה (1234, 0000, ארבע ספרות זהות); (ג) שכחת PIN = שחזור דרך אימות מחדש
בחשבון Google של הבעלים. (ה-PIN הוא שער-הורים בפני הילד, לא אבטחה קריפטוגרפית — ה-hash
מספיק.)

### S3. הסרת מיתוג Minecraft — סיכון קניין רוחני
יש ~31 אזכורי "Minecraft/מיינקראפט" ב-`index.html`, `app.js`, `questions.js`, ותיקיית
`games/`, כולל iframe ל-`classic.minecraft.net`. שימוש בשם ובמותג של Mojang באפליקציה
מסחרית בחנות = דחייה או הסרה בתלונת IP. מה לעשות:
1. לשנות את שם הערכה ל"עולם הבלוקים" (הסגנון הוויזואלי הבלוקי עצמו חוקי — סגנון אמנותי
   אינו מוגן; השם, הלוגו והדמויות המקוריות כן).
2. להסיר מהגרסה החנותית את ה-iframe של `classic.minecraft.net` ולהשאיר משחקים חופשיים
   (ClassiCube שכבר קיים ב-`games/` הוא קוד פתוח — לוודא עמידה ברישיון שלו וייחוס).
3. בשאלות (`questions.js`): לנסח מחדש שאלות שמזכירות Minecraft בשמו לדמויות גנריות
   ("בעולם הבלוקים...").
**בוצע כאשר:** `grep -ri minecraft` על קבצי הפריסה מחזיר 0 תוצאות.

### S4. עמידה במדיניות Families של Google Play
אפליקציה שקהל היעד שלה כולל ילדים חייבת:
1. **הצהרת קהל יעד:** "ילדים ומבוגרים" (ההורה משתמש גם) → נכנסים אוטומטית לדרישות Families.
2. **שער הורים (Parental Gate):** כל קישור חיצוני, מסך הגדרות, והתחברות Google חייבים
   להיות מאחורי אימות מבוגר (ה-PIN מ-S2 עונה על זה — לוודא שגם מסך ה-sign-in הראשוני
   מנוסח כפנייה להורה).
3. **איסוף נתונים מילדים:** Google Sign-In הוא של **ההורה** בלבד; פרופילי הילדים הם שם
   פרטי + נתוני משחק, בלי אימייל/טלפון של הילד. לתעד זאת ב-Data Safety וב-privacy.html
   (קיים — לעדכן תוכן: אילו נתונים, איפה נשמרים, זכות מחיקה).
4. **בלי פרסומות** בשלב ראשון — מפשט דרמטית את האישור. אם בעתיד רוצים מונטיזציה:
   רכישה חד-פעמית/מנוי להורים (מאחורי שער הורים) בטוחה יותר ממודעות.
5. **WebView שטוען תוכן מרוחק:** מותר, אבל התוכן הנטען חייב לעמוד באותה מדיניות —
   עוד סיבה ל-S3 (אין להטעין אתרי צד-שלישי בלתי-מבוקרים בתוך אפליקציית ילדים).

### S5. מחיקת חשבון — דרישת חובה של Google
כל אפליקציה עם יצירת חשבון חייבת לספק: (א) מחיקת חשבון מתוך האפליקציה, (ב) קישור URL
למחיקה גם מחוץ לאפליקציה (מוזן בטופס ב-Play Console). מימוש: כפתור "מחק את החשבון
והמשפחה" במסך ההורה (אחרי PIN + אישור כפול) שמוחק `families/$familyId`, `users/$uid`,
וקורא `user.delete()`; ועמוד `delete-account.html` ב-Hosting עם הוראות (או טופס מחיקה
מקוון עם sign-in).

### S6. אכיפת זמן משחק (AccessibilityService) — לפצל גרסאות
שימוש ב-AccessibilityService למטרה שאינה נגישות נדחה כמעט תמיד ב-Play, וגם
`SYSTEM_ALERT_WINDOW` מעורר בדיקה. פתרון: **שני build flavors** ב-`app/build.gradle`:
- `play`: בלי `GameTimeAccessibilityService`, בלי `SYSTEM_ALERT_WINDOW` — אכיפת הזמן
  נשארת ברמת האפליקציה בלבד (הטיימר הפנימי הקיים ב-web).
- `family`: הגרסה המלאה עם האכיפה הנייטיבית, מופצת ב-sideload כמו היום.
כך אין ויתור על הפיצ'ר בבית ואין סיכון דחייה בחנות.

### S7. הצהרות הרשאות
בגרסת `play` להשאיר רק: INTERNET, CAMERA (עם הצהרת שימוש בטופס: "סריקת קודי QR לאישור
מטלות ע"י ההורה"), POST_NOTIFICATIONS. להסיר מהמניפסט של flavor זה את הרשאות השירותים
המיוחדים. מצלמה חייבת `android:required="false"` (כבר קיים).

### S8a. צ'אט ה-AI (מפתח Groq) בגרסה הציבורית
היום הורה מדביק מפתח Groq משלו בהגדרות. פתרון-ביניים ללא עלות וללא backend: להסתיר את
הפיצ'ר כברירת מחדל בגרסת החנות ולהציג אותו כ"הגדרה מתקדמת להורים" עם הסבר איך משיגים
מפתח חינמי — או להשמיט אותו מגרסת `play` לגמרי. proxy משותף (Cloud Function) דורש
תוכנית Blaze — נדחה עד שיש הכנסות.

### S8. תהליך הפרסום עצמו
1. חשבון מפתח (25$) → 2. `assembleRelease` חתום (AN7) בפורמט **AAB** (`bundleRelease`),
זכור: minify כבוי או בדיקת sideload לפני העלאה (לקח tap_runner) → 3. דף חנות בעברית
(+אנגלית): צילומי מסך, גרפיקה, תיאור ללא הבטחות רפואיות/טיפוליות — לנסח "מותאם לילדים
עם צרכים מיוחדים", לא "טיפול באוטיזם" → 4. שאלון דירוג תוכן → 5. Data Safety → 6.
מסלול Internal testing (עד 100 בודקים, בלי ביקורת מלאה) → 7. Closed testing — **דרישה
חדשה לחשבונות אישיים: 12 בודקים למשך 14 יום לפני production** → 8. Production.
